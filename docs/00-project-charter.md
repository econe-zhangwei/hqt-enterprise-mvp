# 00 Project Charter - 企业端 MVP

## 目的
建立企业端 MVP 的统一范围边界与执行准则，作为产品、设计、前后端、测试的共同依据，避免口头对齐造成偏差。

## 适用范围
适用于“上海惠企政策助理”企业端 MVP（Web H5）阶段。
不包含园区端界面开发，不包含在线申报直连。

## 定义与术语
- MVP：最小可行版本，以“可用、可验证、可迭代”为目标。
- 两段式登录：免登录初筛，查看完整报告或转人工时手机号登录。
- SSOT：Single Source of Truth，单一真相源；本项目接口 SSOT 为 `docs/05-api-contract-v1.md`。
- 可申报：满足政策硬性条件的政策项。
- 可争取：部分条件满足，可通过补齐材料或资质提升后申报。

## 主内容
### 1. 产品目标
1. 让企业快速回答三个问题：我能报什么、我缺什么、该找谁。
2. 建立“匹配 -> 问答 -> 转人工”的服务闭环。
3. 为后续园区端和规模化运营沉淀标准化数据与流程。

### 2. 用户与角色
1. 企业老板/财务/行政：主要使用者。
2. 人工顾问：工单处理与申报辅导执行者。

### 3. MVP 范围（In Scope）
1. 企业信息采集（手填为主）。
2. 政策匹配结果展示与可解释说明。
3. 政策详情查看（原文来源可追溯）。
4. AI 政策问答（有引用、可拒答）。
5. 转人工工单与状态查询。

### 4. MVP 不做项（Out of Scope）
1. 在线申报与政务系统直连。
2. 今日申报独立页面。
3. 政策解读内容栏目、短视频栏目、积分计算器。
4. 园区端页面。
5. 历史申报信息录入。

### 5. 核心指标
1. 可申报政策命中率（人工复核口径）。
2. 完整结果查看率（触发登录后的转化）。
3. 工单提交率与工单完结时长。
4. AI 问答可引用回答占比。

### 6. 团队分工原则
1. 前端负责：页面交互、字段校验、状态展示、工单提交体验。
2. 后端负责：政策库、匹配引擎、问答服务、工单流转、鉴权与审计。
3. 联调依据：`docs/05-api-contract-v1.md` 为唯一接口真相源（SSOT）。

### 7. 文档治理规则
1. 文档采用“临时更新 + 发布前强制校对”。
2. 发布前必须执行 `docs/08-release-checklist.md`。
3. 变更必须同步登记到 `docs/09-changelog.md`。

## 验收标准
1. 团队对 In Scope/Out of Scope 无歧义。
2. 前后端边界明确，接口真相源唯一。
3. 需求评审、技术评审、上线评审可按文档独立执行。
4. 任一成员可仅凭文档理解 MVP 目标和边界。
